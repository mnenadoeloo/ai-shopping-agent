# Product Proposal — AI Shopping Agent

## 1. Обоснование идеи

### Прикладная задача

Пользователи e-commerce платформ сталкиваются с проблемой информационной перегрузки: каталоги содержат десятки тысяч товаров, фильтры не улавливают намерение, а поисковые запросы плохо работают с расплывчатыми или нестандартными описаниями («что подарить другу-геймеру», «ноутбук для работы в кафе»).

Классические рекомендательные системы решают задачу персонализации, но не задачу диалогового уточнения требований. AI Shopping Agent закрывает этот gap: агент понимает намерение на естественном языке, уточняет требования в диалоге и выдаёт обоснованные рекомендации с учётом бюджета, наличия и актуальных цен.

### Целевая аудитория

- Покупатели с нечётким запросом («что-то для подарка», «замена текущему устройству»)
- Покупатели, чувствительные к цене и акциям
- Пользователи, которым важно объяснение рекомендации, а не просто список

## 2. Цель и метрики

### Цель проекта

Продемонстрировать архитектуру агентной LLM-системы для задач e-commerce рекомендаций, способной вести диалог, учитывать контекст и выдавать релевантные, обоснованные результаты.

### Метрики успеха

#### Продуктовые метрики

| Метрика | Описание | Целевое значение (PoC) |
|---|---|---|
| Recommendation Relevance | Доля рекомендаций, оценённых пользователем как релевантные | ≥ 70% |
| CTR (Click-through Rate) | Доля рекомендованных товаров, по которым был совершён переход | ≥ 15% |
| Conversion Proxy | Доля сессий, завершившихся имитацией добавления в корзину | ≥ 5% |
| User Satisfaction | Средняя оценка пользователя по шкале 1–5 | ≥ 3.8 |

#### Агентские метрики

| Метрика | Описание | Целевое значение (PoC) |
|---|---|---|
| Hallucination Rate | Доля ответов, содержащих несуществующие товары или атрибуты | ≤ 3% |
| Intent Recognition Accuracy | Точность распознавания намерения пользователя | ≥ 85% |
| Clarification Rate | Доля сессий, где агент задал уточняющий вопрос вместо пустого ответа | ≥ 90% для ambiguous-запросов |

#### Технические метрики (SLO)

| Метрика | Целевое значение |
|---|---|
| p50 latency (end-to-end) | ≤ 2 с |
| p95 latency (end-to-end) | ≤ 5 с |
| Retrieval recall@10 | ≥ 0.75 |
| Reranking nDCG@5 | ≥ 0.70 |
| Uptime (PoC) | ≥ 95% |

## 3. Сценарии использования

### Основные сценарии

**Сценарий 1 — Подбор по параметрам**
Пользователь: «Подбери ноутбук для программирования до 1500$»
Агент: уточняет предпочтения (ОС, экран, портативность), возвращает топ-3 с объяснением, проверяет наличие и цену в реальном времени.

**Сценарий 2 — Подарочный запрос**
Пользователь: «Что подарить другу, который любит настольные игры?»
Агент: задаёт уточняющие вопросы о бюджете и типе игр, предлагает несколько категорий и конкретные товары.

**Сценарий 3 — Поиск альтернатив**
Пользователь: «Найди похожие товары дешевле»
Агент: использует контекст предыдущего шага, выполняет similarity search с фильтром по цене.

**Сценарий 4 — Диалоговое уточнение**
Пользователь последовательно уточняет запрос через несколько сообщений; агент сохраняет контекст и не переспрашивает уже известное.

### Edge-кейсы

| Ситуация | Ожидаемое поведение агента |
|---|---|
| Слишком общий запрос («что-нибудь интересное») | Задать уточняющий вопрос о категории или бюджете |
| Бюджет ниже минимальной цены в каталоге | Сообщить об ограничении, предложить ближайшие по цене варианты |
| Товар отсутствует в наличии | Предложить альтернативы или уведомить об ожидаемом поступлении |
| Противоречивые требования («самый мощный и самый дешёвый») | Объяснить компромисс, предложить варианты по каждому критерию |
| Неоднозначный запрос («яблоко» — фрукт или бренд?) | Уточнить намерение перед поиском |
| Запрос на несуществующий товар | Не галлюцинировать, сообщить об отсутствии, предложить похожее |
| Инъекция через пользовательский ввод | Санитизировать входные данные, не исполнять инструкции из запроса |

## 4. Ограничения

### Технические ограничения

| Ограничение | Оценка |
|---|---|
| p95 latency | ≤ 5 с (LLM + retrieval + reranking + external API) |
| Размер каталога | ~10k товаров (PoC); масштабирование до 1M требует распределённого индекса |
| Контекстное окно LLM | Ограничено ~128k токенов; длинные диалоги требуют summarization |
| Точность retrieval | Зависит от качества эмбеддингов и размера каталога |
| Зависимость от external API | Нестабильность внешних API влияет на актуальность цен и наличия |

### Операционные ограничения (PoC)

| Ограничение | Оценка |
|---|---|
| Бюджет на LLM API | Ограничен; использование open-source моделей как fallback |
| Инфраструктура | Docker-based, локальный деплой; облачный деплой — следующий этап |
| Команда | Проект реализуется как учебный PoC, без SLA и production SLO |
| Данные | Синтетический датасет; интеграция с реальным каталогом — вне скопа PoC |

## 5. Архитектурный набросок

### Модули системы

```
User Interface (Chat)
        │
        ▼
LLM Orchestrator
  ├── Intent Analyzer
  ├── Tool Selector
  └── Response Generator
              │
   ┌──────────┼──────────────┐
   ▼          ▼              ▼
Product    Reranker   External APIs
Retrieval             ├── Price API
  ├── Embeddings      ├── Availability API
  └── Vector DB       └── Promotion API
        │
        ▼
    Memory
  ├── Dialog History
  ├── User Preferences
  └── Previous Recommendations
```

### Ключевые интеграции

- **Vector DB** (FAISS / Qdrant) — семантический поиск по каталогу товаров
- **HuggingFace Transformers** — эмбеддинги и cross-encoder для reranking
- **OpenAI API / open-source LLM** — оркестрация и генерация ответов
- **Price / Availability / Promotion API** — имитация реальных e-commerce ограничений

## 6. Data Flow и разделение ответственности

### Полный data flow

```
1. Пользователь → User Input (текст)
2. LLM Orchestrator → Intent Analysis (намерение, сущности, бюджет)
3. Orchestrator → Product Retrieval (dense search, top-k кандидатов)
4. Retrieval Results → Reranker (cross-encoder, финальный ранжированный список)
5. Reranked List → External APIs (актуальные цена / наличие / акции)
6. Enriched Results + Dialog History → LLM Response Generator
7. LLM → Сформированный ответ → User Interface
8. Ответ + контекст → Memory (сохранение состояния сессии)
```

### Что делегируется LLM / Agent

| Задача | Делегируется LLM/Agent |
|---|---|
| Анализ намерения пользователя | Да |
| Формулировка уточняющих вопросов | Да |
| Объяснение рекомендаций | Да |
| Обработка edge-кейсов и противоречий | Да |
| Выбор инструментов (retrieval, API) | Да (tool use) |
| Semantic search / эмбеддинги | Нет (отдельная модель) |
| Reranking | Нет (cross-encoder) |
| Хранение и обновление Memory | Нет (внешний модуль) |
| Валидация результатов retrieval | Нет (детерминированная проверка) |
| Запросы к внешним API | Нет (отдельный слой интеграций) |
